# Правила разработки фронтенда

Проверь соответствует ли написанный код правилам ниже. Ты в режиме Агента, поэтому после того, как составишь список исправлений кода, реализуй их сразу!

Придерживайся следующих правил при написании кода:

1. Используй лучшие практики профессиональной разработки: React 19, TypeScript strict, TanStack Query, Tailwind CSS — state of the art для данного стека.

2. Соблюдай слоистую структуру проекта:
   - `api/` — только HTTP-запросы через `apiClient`, никакой логики;
   - `hooks/` — только TanStack Query (`useQuery` / `useMutation`), никаких прямых вызовов `apiClient`;
   - `pages/` — страницы, оркестрируют хуки и компоненты;
   - `components/` — переиспользуемые UI-компоненты (если компонент нужен только в одном файле, держи его там же);
   - `types/` — все TypeScript-интерфейсы, синхронизированные с Pydantic-схемами бэкенда;
   - `lib/` — утилиты и конфиги (`queryClient`, вспомогательные функции).

3. Никогда не вызывай `apiClient` напрямую из компонентов или страниц — только через хуки из `hooks/`.

4. Типизируй всё, где это разумно. Запрещено использовать `any` и приведение `as unknown as X` без крайней необходимости — это признак ошибки в типах данных, которую нужно исправить в источнике.

5. Query keys в TanStack Query должны быть единообразны во всём проекте. Если один и тот же ключ используется в нескольких местах (хук + invalidateQueries) — выноси его в константу или держи в файле хука. Никогда не дублируй строковый литерал ключа в разных файлах.

6. После каждой мутации (create / update / delete) инвалидируй связанные запросы через `qc.invalidateQueries`. Не обновляй кэш вручную через `setQueryData`, если можно просто инвалидировать.

7. Всегда обрабатывай состояния загрузки (`isLoading`) и ошибки (`isError` / `error`) там, где это видит пользователь. Молчаливо проглатывать ошибки запрещено.

8. Денежные значения из API приходят как строки (`Decimal` → `string`). Отображай их всегда через `Number(value).toLocaleString('ru', { minimumFractionDigits: 2 })`. Не используй `toFixed` и не складывай строки чисел напрямую.

9. Строковые константы из API-enum'ов (`'MATCHED'`, `'UNMATCHED'`, `'EXPENSE'` и т.д.) не хардкоди разбросанно по JSX. Выноси соответствия «значение → метка/стиль» в объект-словарь рядом с компонентом или в отдельный файл констант.

10. Формы с валидацией (более одного поля с правилами) реализуй через React Hook Form + Zod. Простые однополевые формы или формы с тривиальными данными — через `useState` допустимо.

11. Декомпозируй компоненты по принципу единственной ответственности. Если страница разрастается, выноси логические секции в именованные подкомпоненты в том же файле или в отдельные файлы `components/`.

12. Не используй «магические числа и строки» — выноси их в именованные константы (`const TOKEN_KEY = 'buh_token'`, `const NAV_ITEMS = [...]`).

13. Один экземпляр `QueryClient` на всё приложение — в `lib/queryClient.ts`. Не создавай новые экземпляры в компонентах или хуках.

14. Покрывай тестами: логику хуков (через `renderHook` + MSW), ключевые сценарии страниц (happy-path + основные ошибки), утилитарные функции. Тесты пиши через Vitest + Testing Library.

15. Поддерживай единый стиль кода через ESLint + Prettier. Запускай проверку перед коммитом.
